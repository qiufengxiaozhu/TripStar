# TripStar 项目配置与启动指南

## 一、环境要求

| 工具 | 版本要求 |
|------|---------|
| Python | 3.10+ |
| Node.js | 18+ |
| npm | 随 Node.js 安装即可 |
| uv | 需安装（`pip install uv`），后端 Agent 运行 MCP 服务时依赖 |

---

## 二、API Key 申请与获取

本项目需要 3 类外部服务的 Key：

### 1. 大模型 API Key（必填）

需要兼容 OpenAI 格式的大模型服务，如 MiniMax、豆包、通义千问、DeepSeek 等。

以 **MiniMax** 为例：
- 注册 [MiniMax 开放平台](https://platform.minimaxi.com/)
- 创建应用 → 获取 API Key
- 国内接口地址：`https://api.minimaxi.com/v1`
- 可用模型：`MiniMax-M2.5`、`MiniMax-M2.7` 等

### 2. 高德地图 Key（必填）

需要申请 **两种类型** 的 Key，来源于同一个高德应用：

- 打开 [高德开放平台控制台](https://console.amap.com/dev/key/app)
- 创建应用后，添加两个 Key：
  - **Web 服务** 类型 → 用于后端调用高德 MCP 服务（搜索景点、查天气等）
  - **Web 端(JS API)** 类型 → 用于前端地图展示，同时会生成一个 **安全密钥(securityJsCode)**

### 3. Unsplash API Key（可选）

用于自动获取城市/景点的高清配图，不配置不影响核心功能。

- 打开 [Unsplash Developers](https://unsplash.com/developers)
- 注册账号 → 创建应用 → 获取 Access Key 和 Secret Key
- 免费额度：Demo 模式每小时 50 次请求，个人使用足够

## 三、配置文件

需要配置 3 个文件：

### 文件 1：`backend/.env`

从模板复制：
```powershell
Copy-Item backend\.env.example backend\.env
```

填入以下内容：

```ini
# ===== 大模型配置（必填）=====
LLM_MODEL_ID=MiniMax-M2.5
LLM_API_KEY=你的大模型API Key
LLM_BASE_URL=https://api.minimaxi.com/v1
LLM_TIMEOUT=120

# ===== 服务器配置 =====
HOST=0.0.0.0
PORT=8000
CORS_ORIGINS=http://localhost:5173,http://localhost:3000
LOG_LEVEL=INFO

# ===== Unsplash 图片服务（可选）=====
UNSPLASH_ACCESS_KEY=你的Unsplash Access Key
UNSPLASH_SECRET_KEY=你的Unsplash Secret Key

# ===== 高德地图（必填，Web服务类型的Key）=====
VITE_AMAP_WEB_KEY=你的高德Web服务Key
```

### 文件 2：`frontend/.env`

从模板复制：
```powershell
Copy-Item frontend\.env.example frontend\.env
```

填入以下内容：

```ini
# 后端API地址（开发环境默认即可）
VITE_API_BASE_URL=http://localhost:8000

# 高德地图Web服务Key（与后端保持一致）
VITE_AMAP_WEB_KEY=你的高德Web服务Key

# 高德地图JS API Key（Web端JS API类型，和上面不是同一个）
VITE_AMAP_WEB_JS_KEY=你的高德JS API Key
```

### 文件 3：`frontend/index.html`

找到 `window._AMapSecurityConfig` 部分，将占位符替换为你的高德安全密钥：

```html
<script type="text/javascript">
  window._AMapSecurityConfig = {
    securityJsCode: '你的高德安全密钥(securityJsCode)'
  }
</script>
```

## 四、PyCharm 启动配置（重要）

### 后端启动配置

在 PyCharm 中创建 **Python** 类型的运行配置（不是 Uvicorn 类型）：

| 字段 | 值 |
|------|---|
| **名称** | server |
| **脚本路径** | `D:\ideaProject\TripStar\backend\run.py` |
| **环境变量** | `PYTHONIOENCODING=utf-8;PYTHONUTF8=1`（见下方说明） |
| **Python 解释器** | `Python 3.12 virtualenv at D:\ideaProject\TripStar\backend\venv` |
| **工作目录** | `D:\ideaProject\TripStar\backend` |

```shell
cd /opt/TripStar/backend
./venv/bin/python3 run.py
```

> **为什么不用 Uvicorn 类型的运行配置？**
> PyCharm 的 Uvicorn 运行配置会把"应用程序文件"和"应用程序名称"拼接在一起，
> 生成类似 `uvicorn backend.run:app.api.main:app` 的错误命令。
> 直接运行 `run.py` 脚本可以避免这个问题，它内部已经写好了 uvicorn 启动逻辑。

> **为什么必须设置 `PYTHONIOENCODING=utf-8;PYTHONUTF8=1`？**
> Windows 默认使用 GBK 编码，uvicorn 的子进程会继承这个编码。
> 当中文城市名（如"北京"）经过子进程时会被 GBK 编码损坏变成 `??`，
> 导致大模型收到的是乱码，无法正常规划行程。
> 设置这两个环境变量可以强制所有 Python 进程使用 UTF-8 编码。

### 前端启动

在 PyCharm 终端中：
```powershell
cd frontend
npm install   # 首次运行
npm run dev
```

或者创建一个 npm 运行配置，脚本选 `dev`，工作目录选 `frontend`。

## 六、踩坑记录与排障经验

### 坑 1：Windows 中文编码损坏（最关键）

**现象**：旅行规划返回的全是"中国-北京景点1"这样的占位数据，后端日志中城市名显示为 `??`。

**原因**：Windows 默认编码是 GBK，uvicorn 用 `--reload` 模式时会 spawn 子进程，子进程的 stdout 继承 GBK 编码。当中文字符串经过子进程传递时被 GBK 编码损坏，大模型收到的是乱码。

**解决**：启动后端时必须设置环境变量：
```
PYTHONIOENCODING=utf-8
PYTHONUTF8=1
```

PyCharm 中在运行配置的"环境变量"字段填入 `PYTHONIOENCODING=utf-8;PYTHONUTF8=1`。

命令行启动则：
```powershell
$env:PYTHONIOENCODING="utf-8"
$env:PYTHONUTF8="1"
python run.py
```

### 坑 2：HelloAgentsLLM 自动检测 Provider 错误

**现象**：后端启动后日志显示 `provider=ollama, base_url=0.0.0.0`，大模型调用全部失败。

**原因**：`HelloAgentsLLM()` 不传参时会自动检测 provider，但它不认识 MiniMax 的 API Key 格式和 base_url。你的 Key 以 `sk-` 开头且长度 > 50，自动检测跳过了它；`minimaxi.com` 也不在它的 URL 匹配规则中。最终 fallback 到了 `ollama`，base_url 误读了 `.env` 中的 `HOST=0.0.0.0`。

**解决**：已在 `llm_service.py` 中修复，显式指定 `provider="custom"` 并手动传入环境变量中的 key/url/model，绕过自动检测。

### 坑 3：MCP 工具名不匹配

**现象**：Agent 日志显示 "未找到工具 amap_maps_text_search"，景点搜索、天气查询全部失败。

**原因**：高德 MCP Server 暴露的工具名是 `maps_text_search`、`maps_weather`（无前缀），MCPTool 注册到 SimpleAgent 时工具名是 `amap`（整体作为一个工具）。但原始提示词要求 Agent 调用 `amap_maps_text_search`，SimpleAgent 的工具注册表里找不到这个名字。

**解决**：已修改提示词，使用 `[TOOL_CALL:amap:tool_name=maps_text_search,...]` 格式。同时创建了 `WrappedMCPTool` 包装类，自动将 SimpleAgent 解析出的平铺参数重组为 MCPTool 期望的嵌套 `arguments` 格式。

### 坑 4：MiniMax 模型不直接输出 JSON

**现象**：行程规划 Agent（STEP 4）不输出 JSON，而是先 `<think>` 思考，然后输出 `<tool_call>` XML 标签想去调天气工具。

**原因**：MiniMax-M2.5 模型有强烈的"先思考后行动"倾向，即使提示词明确说"不要调用工具"，它仍然会在 `<think>` 中决定先查天气。另外天气 API 只能查近几天，查不到两周后的天气，模型认为天气数据不匹配就不愿意直接生成 JSON。

**解决**：
- 行程规划 Agent 改为直接调用 `llm.invoke()` 而不是 `SimpleAgent.run()`（planner 不需要工具）
- 增加最多 3 次重试机制：如果响应中没有 JSON，自动追加"请直接输出 JSON"的指令再调用一次
- 自动去除 `<think>...</think>` 标签后再检查 JSON
- 提示词中明确说明"如果天气日期不匹配，根据季节推测"

### 坑 5：LLM 生成的 JSON 格式错误

**现象**：`json.loads()` 报错 "Expecting ',' delimiter"。

**原因**：LLM 生成的 JSON 可能有语法错误（缺少逗号、字符串未闭合等），尤其是内容较长时。

**解决**：已在 `_parse_response` 中添加 `_fix_json` 方法，尝试修复截断和常见格式问题。如果仍然失败则走 fallback 备用方案。

### 坑 6：max_tokens 不足

**现象**：LLM 响应被截断，JSON 不完整。

**原因**：默认 max_tokens 太小，生成 3 天行程的完整 JSON 需要大量 token。

**解决**：已在 `llm_service.py` 中设置 `max_tokens=8192`。

### 坑 7：PyCharm Uvicorn 运行配置拼接错误

**现象**：`Error loading ASGI app. Attribute "app.api.main:app" not found in module "backend.run".`

**原因**：PyCharm 的 Uvicorn 运行配置会把"应用程序文件"路径转为模块名，和"应用程序名称"拼接，生成 `uvicorn backend.run:app.api.main:app` 这样的错误命令。

**解决**：不使用 Uvicorn 类型的运行配置，改用普通 Python 脚本运行 `run.py`。

### 坑 8：端口被幽灵进程占用

**现象**：`netstat` 显示 8000 端口被某个 PID 占用，但 `taskkill` 提示找不到该进程。

**原因**：Windows 上 uvicorn `--reload` 模式的 reloader 进程被杀后，子进程的端口可能处于 TIME_WAIT 状态，需要等待几十秒自动释放。

**解决**：等待 30-60 秒后端口会自动释放。或者临时换一个端口启动（如 8001）。

## 七、常见问题

**Q: PyCharm / Cursor 报 "无法解析导入 hello_agents"？**
A: 需要将 Python 解释器切换到项目虚拟环境 `backend/venv/Scripts/python.exe`。
- PyCharm：File → Settings → Project → Python Interpreter → 选择 Existing environment → 指向 `backend\venv\Scripts\python.exe`
- Cursor：Ctrl+Shift+P → Python: Select Interpreter → 选择 `backend\venv\Scripts\python.exe`

**Q: 前端地图不显示？**
A: 检查 `frontend/.env` 中的 `VITE_AMAP_WEB_JS_KEY` 和 `frontend/index.html` 中的 `securityJsCode` 是否正确配置。

**Q: 后端启动报 API Key 相关错误？**
A: 确认 `backend/.env` 中的 `LLM_API_KEY`、`LLM_BASE_URL`、`VITE_AMAP_WEB_KEY` 已正确填写。

**Q: 旅行规划返回的全是占位数据（"中国-北京景点1"）？**
A: 检查以下几点：
1. 启动时是否设置了 `PYTHONIOENCODING=utf-8` 和 `PYTHONUTF8=1` 环境变量
2. 后端日志中 LLM 初始化是否显示 `provider=custom`（而不是 `ollama`）
3. 后端日志中城市名是否正常显示中文（而不是 `??`）

**Q: 行程规划耗时很长？**
A: 正常情况下需要 2-5 分钟。流程是：搜索景点 → 查询天气 → 搜索酒店 → 生成行程（可能重试 2-3 次）。每一步都需要调用大模型 + 高德 MCP 服务。
